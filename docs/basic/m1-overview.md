## 1. 자동화가 왜 필요한가요?

### 실제 사례: 송장 처리

어떤 회사의 경리팀이 매일 하는 일을 상상해보세요.

1. 이메일 받기 → 첨부 파일 열기
2. 거래처 이름, 금액, 날짜 확인
3. ERP 시스템에 로그인
4. 데이터 하나씩 입력
5. 저장 → 다음 이메일로 이동
6. **하루 200건 반복**

이 작업을 자동화하면:

| 항목 | 사람이 할 때 | 자동화 후 |
|------|------------|----------|
| 처리 시간 | 연 300시간 | 거의 0 |
| 운영 시간 | 업무 시간만 | 24/7 |
| 오류율 | 피로 누적 시 증가 | 일정 |

{: .note }
> **핵심 메시지:** "사람이 반복하는 일은 기계가 할 수 있습니다."  
> 자동화의 목적은 사람을 대체하는 것이 아니라, 사람이 더 가치 있는 일에 집중하도록 돕는 것입니다.

## 2. Power Platform 구성 요소

Microsoft Power Platform은 4가지 도구로 이루어져 있습니다.

| 도구 | 역할 |
|------|------|
| **Power BI** | 데이터 시각화 및 보고서 |
| **Power Apps** | 기업용 앱 개발 |
| **Power Automate** | 업무 자동화 (클라우드 + RPA) |
| **Copilot Studio** | AI 챗봇 만들기 |

{: .note }
> **이 과정의 범위:** Power Automate(PA)와 Power Automate Desktop(PAD)에 집중합니다.

## 3. PA vs PAD — 어떻게 다른가요?

| 구분 | Power Automate (PA) | Power Automate Desktop (PAD) |
|------|---------------------|------------------------------|
| **목적** | 클라우드 서비스 간 통합 | PC 반복 작업 자동화 |
| **기술** | API / 커넥터 | UI 자동화 / RPA |
| **실행 위치** | 클라우드 서버 | 내 PC |
| **비유** | 시스템끼리 대화 | 사람 손을 대신 |
| **예시** | 이메일 받으면 Teams 알림 | 웹사이트에서 데이터 긁어서 엑셀에 기록 |

{: .note }
> **핵심 메시지:**  
> "PA는 시스템이 연결된 자동화, PAD는 사람 손을 대신하는 자동화"

## 4. 테넌트 / 환경 개념

### 테넌트(Tenant)
우리 회사 전용 Microsoft 클라우드 공간입니다.  
`@회사이름.com` 계정으로 로그인하면 그 회사의 테넌트에 접속하는 것입니다.

### 환경(Environment)
테넌트 안에서 개발/테스트/운영을 구분하는 공간입니다.

```
[테넌트: company.com]
  ├── 기본 환경 (Default) ← 오늘 실습에서 사용
  ├── 개발 환경 (Dev)
  └── 운영 환경 (Prod)
```

{: .note }
> 오늘 실습은 **기본 환경(Default Environment)**에서 진행합니다.

## 5. Attended vs Unattended

| 구분 | Attended | Unattended |
|------|----------|------------|
| **의미** | 내가 보는 앞에서 실행 | 혼자 알아서 실행 |
| **예시** | 내 PC에서 직접 버튼 눌러 시작 | 서버에서 새벽 2시에 자동 실행 |
| **라이선스** | 기본 포함 | 추가 라이선스 필요 |
| **오늘 실습** | ✅ 이것만 사용 | — |

{: .warning }
> **Unattended 자동화**는 Windows Home과 기본 Microsoft 365 라이선스에서는 지원되지 않습니다. 오늘은 Attended 모드만 사용합니다.

---

## ✅ 핵심 정리

- 자동화 = 반복 작업을 기록해서 기계가 대신 실행
- PA = 클라우드 서비스 연결 / PAD = 내 PC의 UI 자동화
- 테넌트 = 회사 전용 MS 클라우드 / 환경 = 그 안의 구분 공간
- 오늘 실습 = Attended 모드 (내 PC에서 직접 실행)
