# T1. LLM 프롬프트 가이드

PAD/PA 작업 중 LLM을 효과적으로 활용하는 방법을 정리합니다.

---

## 나쁜 프롬프트 vs 좋은 프롬프트

| ❌ 나쁜 예 | ✅ 좋은 예 |
|-----------|----------|
| "전화번호 표현식 알려줘" | "PAD에서 핸드폰번호 형식(010-XXXX-XXXX) 유효성 검사하는 표현식" |
| "날짜 바꾸는 방법" | "PAD에서 CurrentDateTime을 yyyy-MM-dd 텍스트로 변환하는 표현식" |
| "조건문 예시" | "PAD에서 텍스트 변수가 비어있는지 확인하는 If 조건 작성법" |
| "셀렉터 찾아줘" | "이 HTML 코드에서 PAD 웹자동화에 쓸 최적의 CSS 셀렉터 찾아줘: [코드]" |
| "For Each 어떻게 써요" | "PAD에서 DataTable을 For Each로 반복할 때 특정 열 값을 가져오는 표현식" |
| "이메일 보내는 법" | "PAD에서 Outlook으로 이메일 보낼 때 본문에 변수 값을 포함하는 방법" |

---

## 좋은 프롬프트의 3가지 법칙

```
도구 명시 (PAD / PA / Copilot Studio)
+ 변수 타입 또는 현재 상황 명시
+ 원하는 결과 명시
= 쓸만한 답변
```

### 법칙 1. 도구를 명시하세요

❌ "날짜 형식 바꾸는 표현식"  
✅ "**PAD**에서 날짜 형식 바꾸는 표현식"

PAD와 PA의 표현식 문법은 다릅니다. 도구를 명시해야 맞는 답이 나옵니다.

### 법칙 2. 현재 상황을 포함하세요

❌ "엑셀 읽는 법"  
✅ "PAD에서 엑셀 파일 전체를 DataTable로 읽은 후 특정 열의 값을 꺼내는 방법"

현재 어떤 변수 타입인지, 어디서 막혔는지 포함하면 훨씬 정확한 답이 나옵니다.

### 법칙 3. 코드/셀렉터는 직접 붙여넣으세요

❌ "셀렉터가 안 되는데 어떻게 해요"  
✅ "이 CSS 셀렉터가 PAD에서 동작하지 않습니다. 수정해줘: `div:nth-child(3) > span`"

실제 코드를 붙여넣으면 즉시 수정 가능한 답변을 받을 수 있습니다.

## 상황별 프롬프트 템플릿

### 표현식이 필요할 때
```
PAD에서 [상황 설명]을 처리하는 표현식을 알려줘.
현재 변수: [변수명] (타입: [텍스트/숫자/DateTime 등])
원하는 결과: [결과 설명]
```

### 셀렉터가 필요할 때
```
이 CSS 셀렉터를 PAD 웹자동화에 최적화해줘.
id나 의미있는 class를 우선 사용하도록 단순화해줘.
원본 셀렉터: [붙여넣기]
```

### 오류가 발생했을 때
```
PAD에서 다음 오류가 발생했습니다:
오류 메시지: [오류 내용 붙여넣기]
액션: [어떤 액션을 실행 중이었는지]
해결 방법을 알려줘.
```

### 흐름 설계가 막힐 때
```
PAD로 다음 작업을 자동화하려고 합니다:
[원하는 자동화 설명]
서브흐름 구조와 필요한 액션 목록을 알려줘.
```

## LLM 활용 시 주의사항

{: .warning }
> **LLM 답변은 반드시 직접 테스트하세요.**  
> LLM이 제시하는 표현식이나 셀렉터가 항상 정확하지는 않습니다.  
> 복사해서 PAD에 붙여넣고 실행해서 직접 확인하는 것이 필수입니다.

{: .note }
> **잘못된 답이 나왔을 때:** 오류 메시지를 그대로 LLM에게 다시 붙여넣으세요.  
> "이렇게 했더니 이런 오류가 났어. 수정해줘."라고 이어서 질문하면 됩니다.
